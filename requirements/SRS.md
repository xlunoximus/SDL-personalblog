# 1 Введение

<a name="appointment"/>

## 1.1 Назначение
В этом документе описаны функциональные и нефункциональные требования к веб-приложению «PersonalBlog». Этот документ предназначен для команды разработчиков, тестировщиков и других заинтересованных сторон, участвующих в создании и проверке корректности работы приложения.

<a name="business_requirements"/>

## 1.2 Бизнес-требования

<a name="initial_data"/>

### 1.2.1 Исходные данные
В современном мире люди всё чаще ведут личные блоги для обмена мыслями, опытом и контентом. Текущее приложение на React и Spring Boot предоставляет базовую функциональность для создания и просмотра постов, но требует формализации требований для дальнейшего развития. «PersonalBlog» решает проблему хранения и обмена личным контентом, предоставляя безопасный и удобный интерфейс для авторов и читателей.

<a name="business_opportunities"/>

### 1.2.2 Возможности бизнеса
«PersonalBlog» ориентировано на индивидуальных пользователей: блогеров, авторов контента, студентов и энтузиастов. Платформа предоставляет:

Аутентификацию для безопасного доступа.
Создание, просмотр, редактирование и удаление постов.
Добавление и просмотр комментариев для взаимодействия.
Хранение данных в PostgreSQL для持久ности.
Приложение подходит как для личного использования, так и для малого сообщества, обеспечивая простой интерфейс для обмена идеями.

<a name="project_boundary"/>

### 1.2.3 Границы проекта
Платформа «PersonalBlog» предоставляет инструменты для создания и управления личными постами, включая аутентификацию, комментарии и базовый просмотр. Приложение поддерживает веб-доступ, но не включает функции социальной сети, монетизации или интеграции с внешними сервисами (например, социальными сетями).

<a name="analogues"/>

## 1.3 Аналоги
Вдохновением для «PersonalBlog» послужили платформы Medium и WordPress. Medium предоставляет простой интерфейс для публикации статей, а WordPress — гибкую систему управления контентом. «PersonalBlog» объединяет их преимущества, добавляя:

Интеграцию frontend (React) и backend (Spring Boot).
JWT-аутентификацию для безопасности.
REST API для обмена данными.
Это делает платформу простым инструментом для личного блогинга.

<a name="user_requirements"/>

# 2 Требования пользователя

<a name="program_interfaces"/>

## 2.1 Программные интерфейсы

Система взаимодействует со следующими внешними системами, библиотеками и сервисами:

- Frontend: React.js (для создания пользовательского интерфейса), Axios (для HTTP-запросов к backend).
- Backend: Spring Boot (фреймворк для Java-приложений), Spring Security (для аутентификации и авторизации), Spring Data JPA (для работы с БД).
- База данных: PostgreSQL (реляционная СУБД для хранения данных о пользователях, постах и комментариях).
- Другие: JWT (JSON Web Tokens) для токенов аутентификации, REST API для обмена данными между frontend и backend. Нет взаимодействия с внешними сервисами вроде облачных API (например, AWS или Google), все локально.

<a name="user_interface"/>

## 2.2 Интерфейс пользователя

Система взаимодействует с пользователем через веб-браузер с использованием React-компонентов. Интерфейс включает навигационную панель (Home, Login/Register, My Posts), формы для ввода данных и списки для просмотра.

Текстовое описание:

- Главная страница: Список всех постов в виде карточек (заголовок, анонс текста, дата, автор). Клик на карточку открывает полный пост.
- Страница поста: Полный текст поста, раздел комментариев с формой добавления комментария (для авторизованных пользователей).
- Страница аутентификации: Формы для регистрации (email, password, username) и логина (email, password).
- Страница управления постами (для автора): Список своих постов с кнопками редактирования и удаления, форма создания нового поста (заголовок, текст).

<a name="user_specifications"/>

## 2.3 Характеристики пользователей

<a name="user_classes"/>

### 2.3.1 Классы пользователей

| Класс пользователей | Описание |
|:---|:---|
|Авторы|Основные пользователи, ведущие блог. Обладают базовыми навыками работы с веб-приложениями. Уровень технической грамотности: средний|
|Читатели|Любые пользователи, просматривающие контент. Минимальный опыт. Уровень технической грамотности: низкий-средний|

<a name="application_audience"/>

### 2.3.2 Целевая аудитория

Основная целевая аудитория:

1. Авторы блогов и энтузиасты (70%)
2. Читатели и случайные посетители (30%)

<a name="assumptions_and_dependencies"/>

## 2.4 Предположения и зависимости

Доступ к интернету и современному браузеру (Chrome, Firefox) для frontend. 
Установленная PostgreSQL для backend (локально или на сервере). 
Java 17+ и Node.js для разработки/запуска. 
Требования могут измениться при добавлении новых фич (например, интеграция с социальными сетями), или если изменится стек технологий. 
Зависимость от стабильности БД: если PostgreSQL недоступна, приложение не запустится. 
Предполагается, что пользователи имеют email для регистрации.

# 3 Системные требования

<a name="functional_requirements"/>

## 3.1 Функциональные требования

<a name="authentication"/>

### 3.1.1 Аутентификация пользователей

<a name="user_registration"/>

#### 3.1.1.1 Регистрация пользователя

|Функция|Требования|
|:---|:---|
|Ввод данных|Система должна принимать email, username, password|
|Валидация|Проверка уникальности email и username, минимальная длина пароля (8 символов)|
|Хранение|Данные хранятся в PostgreSQL с хэшированием пароля (BCrypt)|
|Обработка ошибок|При дубликате — уведомление пользователя|

<a name="user_login"/>

#### 3.1.1.2 Вход в систему

|Функция|Требования|
|:---|:---|
|Ввод credentials|Система принимает email и password|
|Аутентификация|Проверка через Spring Security, выдача JWT-токена|
|Сессия|Токен хранится в localStorage frontend|
|Обработка ошибок|Неверные данные — 401 Unauthorized|

<a name="post_management"/>

### 3.1.2 Управление постами

<a name="create_post"/>

#### 3.1.2.1 Создание поста

|Функция|Требования|
|:---|:---|
|Ввод данных|Заголовок, текст (авто-дата)|
|Авторизация|Только авторизованные пользователи|
|Сохранение|POST /api/posts, хранение в БД|
|Валидация|Не пустой заголовок и текст|

<a name="view_posts"/>

#### 3.1.2.2 Просмотр постов

|Функция|Требования|
|:---|:---|
|Список постов|GET /api/posts — все посты|
|Отдельный пост|GET /api/posts/{id} — с комментариями|
|Доступ|Доступно всем|

<a name="edit_delete_posts"/>

#### 3.1.2.3 Редактирование и удаление постов

|Функция|Требования|
|:---|:---|
|Редактирование|PUT /api/posts/{id} — только автор|
|Удаление|DELETE /api/posts/{id} — только автор|
|Авторизация|Проверка через JWT|

<a name="comment_management"/>

### 3.1.3 Управление комментариями

<a name="add_comment"/>

#### 3.1.3.1 Добавление комментария

|Функция|Требования|
|:---|:---|
|Ввод текста|POST /api/comments/{postId}|
|Авторизация|Только авторизованные|
|Сохранение|Хранение в БД, связь с постом|

<a name="restrictions_and_exclusions"/>

### 3.1.4 Ограничения и исключения

1. Система не поддерживает загрузку изображений в посты в текущей версии
2. Нет функций модерации комментариев
3. Система не интегрируется с внешними социальными сетями
4. Нет уведомлений о новых комментариях
5. Зависимость от БД: без PostgreSQL система не запускается

<a name="non-functional_requirements"/>

## 3.2 Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1 Атрибуты качества

<a name="reliability"/>

#### 3.2.1.1 Надежность

Важность: Критична для сохранения данных.
Требования:

1. Система должна восстанавливаться после сбоев БД (retry mechanisms)
2. MTBF > 100 часов
3. Транзакции для операций с БД
4. Логирование ошибок

<a name="security"/>

#### 3.2.1.2 Безопасность

Важность: Высокая, для защиты пользовательских данных.
Требования:

1. Защита от SQL-инъекций (JPA)
2. Хранение паролей хэшированными (BCrypt)
3. JWT для авторизации
4. HTTPS для production
5. Соблюдение GDPR

<a name="usability"/>

#### 3.2.1.3 Удобство использования

Важность: Высокая, для привлечения пользователей.
Требования:

1. Интуитивный UI
2. Время обучения < 5 мин
3. Измерение через user feedback

<a name="maintainability"/>

#### 3.2.1.4 Поддерживаемость

Важность: Критична для обновлений.
Требования:

1. Модульная структура
2. Документация API (Swagger)
3. Unit-тесты (coverage > 70%)
4. Git для версионного контроля

<a name="efficiency"/>

#### 3.2.1.5 Эффективность

Важность: Для хостинга.
Требования:

1. CPU < 50%, memory < 512MB
2. Мониторинг производительности

<a name="restrictions"/>

### 3.2.2 Ограничения

1. Язык: Java для backend, JS для frontend
2. Система сборки: Maven/Gradle для backend, npm для frontend
3. БД: PostgreSQL
4. Фреймворк: Spring Boot, React
5. Формат данных: JSON
6. Протокол: HTTPS в production
7. Зависимость от БД и интернета
